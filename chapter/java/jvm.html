1. 运行时数据区域

1.1. 程序计数器

记录正在执行的虚拟机字节码指令的地址。（若正在执行的是本地方法则为空）

特点：线程私有、无OutOfMemoryError。

1.2. 虚拟机栈

每个 Java 方法在执行的同时会创建一个栈帧用于存储局部变量表、操作数栈、动态链接、方法出口等信息。从方法调用直至执行完成的过程，就对应着一个栈帧在 Java 虚拟机栈中入栈和出栈的过程。

特点：线程私有、StackOverflowError、OutOfMemoryError

1.3. 本地方法栈

与 Java 虚拟机栈类似，只不过是本地方法栈为本地方法服务。

特点：线程私有、StackOverflowError、OutOfMemoryError

1.4. 堆

所有对象实例和数组都在这里分配内存，是垃圾收集的主要区域。

特点：线程共享、OutOfMemoryError

1.5. 方法区

用于存放已被加载的类信息、常量、静态变量、即时编译器编译后的代码等数据（klass元信息、运行时常量池）。

特点：线程共享、OutOfMemoryError

1.6. 运行时常量池

是方法区的一部分。存储编译期生成的字面量和符号引用（允许动态生成）。

2. 类加载机制

2.1. 类加载过程

包含了加载、验证、准备、解析和初始化这 5 个阶段。

  • 加载：将字节码文件读入内存，并在内存中生成一个代表该类的 Class 对象。作为方法区中该类各种数据的访问入口。
  • 验证：校验字节码文件的正确性。
  • 准备：给类的静态变量分配内存并设置默认值。
  • 解析：将常量池中符号引用替换为直接引用。
  • 初始化：对类的静态变量初始化为指定的值，执行静态代码块。

2.2. 类加载机制

2.2.1. 类加载器

类加载器种类：

  • 启动类加载器：负责加载 JRE 的 lib 目录下的核心类库，比如 rt.jar 等。
  • 扩展类加载器：负责加载 JRE的 lib 目录下的 ext 扩展目录中的 jar 包。
  • 应用程序类加载器：负责加载 ClassPath 路径下的类包。
  • 自定义类加载器：负责加载用户自定义路径下的类包。

类加载器初始化过程：

首先创建 JVM 启动器实例 sun.misc.Launcher，Launcher 初始化使用了单例模式设计，保证一个 JVM 虚拟机内只有一个 Launcher 实例。

在 Launcher 构造方法内部，其创建了两个类加载器，分别是 ExtClassLoader（扩展类加载器）和 AppClassLoader（应用程序类加载器）。Launcher 默认返回 AppClassLoader 实例。源码如下：

code 
public Launcher() {
    Launcher.ExtClassLoader var1;
    try {
        //构造扩展类加载器，在构造的过程中将其父加载器设置为null。
        var1 = Launcher.ExtClassLoader.getExtClassLoader();
    } catch (IOException var10) {
        throw new InternalError("Could not create extension class loader", var10);
    }

    try {
        //构造应用类加载器，在构造的过程中将其父加载器设置为ExtClassLoader，
        //Launcher的loader属性值是AppClassLoader，我们一般都是用这个类加载器来加载我们自己写的应用程序
        this.loader = Launcher.AppClassLoader.getAppClassLoader(var1);
    } catch (IOException var9) {
        throw new InternalError("Could not create application class loader", var9);
    }

    Thread.currentThread().setContextClassLoader(this.loader);
    String var2 = System.getProperty("java.security.manager");
    ...//省略一些不需关注代码
}

2.2.2. 双亲委派机制

一个类加载器首先将类加载请求转发到父类加载器，只有当父类加载器无法完成时才尝试自己加载。

好处：

  • 沙箱安全机制：自己写的 java.lang.String 类不会被加载，这样便可以防止核心API库被随意篡改
  • 避免类的重复加载：当父亲已经加载了该类时，就没有必要子 ClassLoader 再加载一次，保证被加载类的唯一性

实现：

双亲加载机制实现于 ClassLoader 的 loadClass() 方法中。先检查类是否已经加载过，如果没有则让父类加载器去加载。当父类加载器加载失败时抛出 ClassNotFoundException，此时尝试自己去加载。源码如下：

code 
protected Class<?> loadClass(String name, boolean resolve)
    throws ClassNotFoundException
{
    synchronized (getClassLoadingLock(name)) {
        // 检查当前类加载器是否已经加载了该类
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            long t0 = System.nanoTime();
            try {
                if (parent != null) {  //如果当前加载器父加载器不为空则委托父加载器加载该类
                    c = parent.loadClass(name, false);
                } else {  //如果当前加载器父加载器为空则委托引导类加载器加载该类
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
            }

            if (c == null) {
                long t1 = System.nanoTime();
                //都会调用URLClassLoader的findClass方法在加载器的类路径里查找并加载该类
                c = findClass(name);
                sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                sun.misc.PerfCounter.getFindClasses().increment();
            }
        }
        if (resolve) {  //不会执行
            resolveClass(c);
        }
        return c;
    }
}

3. 垃圾回收机制

3.1. 内存分配策略

3.1.1. 对象栈上分配

我们知道对象是在堆上进行分配。当JVM通过逃逸分析确定该对象会不会被外部访问。如果不会逃逸可将该对象在栈上分配内存，这样该对象所占用的内存空间就可以随栈帧出栈而销毁，就减轻了垃圾回收的压力。

标量与聚合量：

标量即不可被进一步分解的量，而 JAVA 的基本数据类型就是标量（如：int，long等基本数据类型以及 reference 类型等），标量的对立就是可以被进一步分解的量，而这种量称之为聚合量。而在 JAVA 中对象就是可以被进一步分解的聚合量。

标量替换：

通过逃逸分析确定该对象不会被外部访问，并且对象可以被进一步分解时，JVM不会创建该对象，而是将该对象成员变量分解若干个被这个方法使用的成员变量所代替，这些代替的成员变量在栈帧或寄存器上分配空间，这样就不会因为没有一大块连续空间导致对象内存不够分配。

3.1.2. TLAB

堆是一个线程共享区域，用于分配对象，当多个线程同时请求分配对象时，就需要进行同步，使得效率变差。

TLAB(Thread Local Allocation Buffer)即线程本地分配缓存。使用TLAB，就可以每个线程在分配对象到堆空间时，先分配到自己所属的那一块堆空间中，避免同步带来的效率问题，从而提高分配效率。

3.1.3. 对象在Eden区分配

大多数情况下，对象在新生代 Eden 区分配，当 Eden 区空间不够时，发起 Minor GC。剩余存活的对象会被挪到为空的那块 Survivor 区，下一次Eden区满了后又会触发 Minor gc ，把eden区和survivor区垃圾对象回收，把剩余存活的对象挪到另外一块空的 Survivor 区。

新生代的对象都是朝生夕死的，存活时间很短。所以尽量让 Eden 区更大，Survivor 区够用即可。Eden 与 Survivor 区默认8:1:1。

3.1.4. 大对象直接进入老年代

大对象就是需要大量连续内存空间的对象（比如：字符串、数组），可通过 JVM 参数设置大对象的大小，超过该值直接在老年代分配，避免 Eden 区和 Survivor 区大量内存复制。

3.1.5. 长期存活的对象进入老年代

为对象定义年龄计数器，对象在 Eden 出生并经过 Minor GC 依然存活，将移动到 Survivor 中，年龄就增加 1 岁，增加到一定年龄则移动到老年代中。可通过 JVM 参数设置年龄阈值。

3.1.6. 对象动态年龄判断

如果在 Survivor 中相同年龄所有对象大小的总和大于 Survivor 空间的一半，则年龄大于或等于该年龄的对象可以直接进入老年代。

3.2. 堆内存的回收

3.2.1. 引用类型

  • 强引用

    使用 new 一个新对象的方式创建的引用，只要强引用还在，永远不会回收被引用的对象。

  • 软引用

    用来描述一些有用但非必需的对象。使用 SoftReference 类来创建，被软引用关联的对象只有在内存不够的情况下才会被回收。

    可用于实现缓存（内存不足才回收）

  • 弱引用

    也是用来描述非必需对象，使用 WeakReference 类来创建。被弱引用关联的对象一定会被回收，也就是说它只能存活到下一次垃圾回收发生之前。

  • 虚引用

    使用 PhantomReference 来创建虚引用。一个对象是否有虚引用的存在，不会对其生存时间造成影响，也无法通过虚引用得到一个对象。

    为一个对象设置虚引用的唯一目的是能在这个对象被回收时收到一个系统通知。

3.2.2. 判断一个对象是否可回收

  • 引用计数法

    为对象添加一个引用计数器，当对象增加一个引用时计数器加 1，引用失效时计数器减 1。引用计数为 0 的对象可被回收。

    缺点：出现循环引用时，对象永远得不到回收。

  • 可达性分析法

    以 GC Roots 为起始点进行搜索，可达的对象都是存活的，不可达的对象可被回收。Java 虚拟机使用此方法判断对象是否可回收。

    GC Roots 的对象一般包含以下内容：

    • 虚拟机栈中局部变量表中引用的对象
    • 本地方法栈中 JNI(Native方法) 引用的对象
    • 方法区中类静态属性引用的对象
    • 方法区中的常量引用的对象

3.2.3. finalize() 方法

当一个对象可被回收时，如果需要执行该对象的 finalize() 方法，那么就有可能在该方法中让对象重新被引用，从而实现自救。自救只能进行一次。

该方法代价大，可用 try-finally 等方式替代，不推荐使用。

3.2.4. 垃圾回收算法

标记-清除 (Mark-Sweep)

标记要回收的对象，然后清除。

缺点：

  • 标记与清除过程效率不高
  • 产生内存碎片，导致大对象无法分配

标记-整理 (Mark-Compact)

让所有存活的对象都向一端移动，然后直接清理掉端边界以外的内存。

复制

将内存划分为大小相等的两块，每次只使用其中一块，当这一块内存用完了就将还存活的对象复制到另一块上面，然后再把使用过的内存空间进行一次清理。

HotSpot 虚拟机的 Eden 和 2 * Survivor 大小比例默认为 8:1:1，将一份 Survivor 用于复制，保证内存利用率达90%。

缺点：

  • 内存浪费
  • 用于复制的内存 (Survivor) 不足时，需要老年代进行分配担保（放不下的对象存入老年代）

3.2.5. 分代收集

新生代和老年代

  • 新生代每次垃圾回收时只有少量对象存活，故适用复制算法
  • 老年代对象存活率高，故适用标记-清除或标记-整理算法

Minor GC 和 Full GC

  • Minor GC：回收新生代上，因为新生代对象存活时间很短，因此 Minor GC 会频繁执行，执行的速度一般也会比较快。
  • Full GC：回收老年代和新生代，老年代对象其存活时间长，因此 Full GC 很少执行，执行速度会比 Minor GC 慢很多。

空间分配担保

在发生 Minor GC 之前，虚拟机先检查老年代最大可用的连续空间是否大于新生代所有对象总空间。

  • 若成立，则认为 Minor GC 是安全的，进行一次Minor GC
  • 若不成立，虚拟机会查看 HandlePromotionFailure 设置值是否允许担保失败
    • 若允许则继续检查老年代最大可用的连续空间是否大于历次晋升到老年代对象的平均大小
      • 如果大于，进行一次 Minor GC
      • 如果小于，进行一次 Full GC
    • 若不允许则进行 Full GC

Full GC 的触发条件

对于 Minor GC，其触发条件非常简单，当 Eden 空间满时，就将触发一次 Minor GC。而 Full GC 则相对复杂，有以下条件：

  • 调用System.gc()
  • 老年代空间不足
  • 空间分配担保失败
  • JDK1.7 及以前的永久代空间不足
  • CMS GC 过程发生 Concurrent Mode Failure

3.2.6. 垃圾回收器

垃圾回收器是内存回收的具体实现。

相关指标：

  • 吞吐量 = 用户线程用时 / 程序总用时

Serial 收集器

单线程、串行收集器。GC 时会 "Stop The World"。新生代采用复制算法，老年代（Serial Old）采用标记-整理算法。

特点：简单高效

Parallel Scavenge收集器

多线程收集器。新生代采用复制算法，老年代（Parallel Old）使用标记整理算法。具有 GC 自适应调节策略。

特点：多线程、注重吞吐量（CPU 利用率）

ParNew 收集器

Serial 收集器的多线程版本。与 Parallel 类似，区别在于 ParNew 可以与 CMS 配合使用。

特点：多线程、可与 CMS 配合使用

CMS 收集器

CMS (Concurrent Mark Sweep)，并发收集器，Mark Sweep 指标记清除算法。是一种以获取最短回收时间停顿为目标的收集器。

回收流程：

  • 初始标记：只标记 GC Roots 能直接引用的对象，速度很快，需要停顿。
  • 并发标记：进行 GC Roots Tracing 的过程，GC 线程与用户线程并发执行。在整个回收过程中耗时最长，不需要停顿。
  • 重新标记：为了修正并发标记期间因用户程序继续运作而导致标记产生变动的那一部分对象的标记记录，需要停顿。
  • 并发清除：不需要停顿。

特点：

  • 并发收集：用户线程暂停时间短。
  • 吞吐量低：线程频繁切换，导致 CPU 利用率不高。
  • 产生浮动垃圾：指并发清除阶段由于用户线程继续运行而产生的垃圾，这部分垃圾只能到下一次 GC 时才能进行回收。由于浮动垃圾的存在，因此需要预留出一部分内存，意味着 CMS 收集不能像其它收集器那样等待老年代快满的时候再回收。如果预留的内存不够存放浮动垃圾，就会出现 Concurrent Mode Failure，这时虚拟机将临时启用 Serial Old 来替代 CMS。
  • 空间碎片：标记 - 清除算法导致空间碎片。

三色标记算法

CMS 在并发标记过程中，使用的标记算法为三色标记算法。算法将所有对象标记成三种颜色：

  • 黑色：自身与引用的对象（成员变量）均被访问过。
  • 灰色：自身被访问过，至少存在一个引用的对象没被访问过。
  • 白色：对象没被访问过，表示不可达。

多标：

在并发标记过程中，若由于方法运行结束导致部分局部变量（GCRoot）被销毁，这个GCRoot引用的对象之前又被扫描过(被标记为非垃圾对象)，那么本轮GC不会回收这部分内存。称之为“浮动垃圾”，下一轮垃圾回收中才被清除。

另外，针对并发标记（还有并发清除）开始后产生的新对象，通常的做法是直接全部当成黑色，本轮不会进行清除。这部分对象期间可能也会变为垃圾，这也算是浮动垃圾的一部分。

漏标：

漏标会导致被引用的对象被当成垃圾误删除。

产生原因：1、黑色对象指向了白色对象，2、并且同时，灰色对象取消了白色对象的引用。

因为黑色对象已经标记完毕，不会再去扫描，并且也没有灰色对象的指向，故产生漏标。

解决方案： 增量更新（Incremental Update） 和原始快照（Snapshot At The Beginning，SATB）。

  • 增量更新

    关注引用的增加，若发现黑色指向了白色，就将这个新插入的引用记录下来， 并发扫描结束后， 将这些记录过的引用重新扫描一次。 可以简化理解为， 黑色对象一旦新插入了指向白色对象的引用之后， 它就变回灰色对象了。（CMS 解决漏标方式，通过写后屏障实现）

  • 原始快照

    关注引用的删除，若发现灰色对象对白色对象的引用删除，就将这个要删除的引用记录下来， 在并发扫描结束后， 将这些记录过的引用重新扫描一次，这样就能扫描到白色的对象。（G1 解决漏标方式，通过写前屏障实现）

增量更新与原始快照都是通过写屏障实现。所谓写屏障，是指在赋值操作前后，加入特殊处理（类似 AOP 的思想）。

记忆集与卡表

  • 记忆集

    在可达性分析过程中存在跨代引用情况。在收集区域引入记忆集（Remember Set）数据结构（记录从非收集区到收集区的指针集合），避免把非收集区域加入 GCRoots 扫描范围。

    收集器只需通过记忆集判断出某一块非收集区域是否存在指向收集区域的指针即可，无需了解跨代引用指针的全部细节。

  • 卡表

    Hotspot 使用卡表（Cardtable）来实现记忆集（抽象概念）。

    卡表内部为一个字节数组：CARD_TABLE，每个元素对应着其标识的内存区域一块特定大小的内存块，称为“卡页”。HotSpot使用的卡页是 2^9 大小，即 512 字节。

    若某个卡页中存在指向回收区的指针，对应的卡表标识为 1，称这个元素为 Dirty。否则为 0。GC 时，只要筛选本收集区的卡表中 Dirty 元素对应卡页脏对象的引用加入 GCRoots 中。

    Hotspot 使用写屏障维护卡表状态。

G1 收集器

G1 把堆划分成多个大小相等的独立区域 Region。保留了年轻代和老年代的概念，但不再是物理隔阂。与其他收集器不同，G1有专门分配大对象的 Region 叫 Humongous 区，而不是让大对象直接进入老年代的 Region 中。

大对象的判定规则就是一个对象是否超过一个 Region 大小的 50%。

GC 步骤：

  • 初始标记：同 CMS 初始标记。
  • 并发表集：同 CMS 并发标记。
  • 最终标记：同 CMS 重新标记。(使用增量更新解决漏标问题)
  • 筛选回收：首先对各个 Region 中的回收价值和成本进行排序，根据用户所期望的 GC 停顿时间（STW）来制定回收计划（CMS 为并发回收）。在 Region 之间采用复制算法回收。

特点：

  • 空间整合：整体上使用标记整理算法，局部（两个 Region 之间）使用复制算法实现的。
  • 可预测停顿：用户可指定停顿时间。

收集分类：

  • YoungGC：当Eden区放满，G1会计算此时回收 Eden 区所需时间。若回收时间远小于参数 -XX:MaxGCPauseMills 设定的值，则增加年轻代的 Region ，继续给新对象存放，直到某次Eden区放满。G1计算回收时间接近参数 -XX:MaxGCPauseMills 设定的值，才会触发Young GC。
  • MixedGC：老年代的堆占有率达到参数 -XX:InitiatingHeapOccupancyPercent 设定的值则触发，回收所有的 Young 和部分 Old 以及 Humongous。使用复制算法，把各 Region 中存活的对象拷贝到别的 Region 中，拷贝过程中若发现没有足够的空 Region 能够承载拷贝对象则触发一次 Full GC。
  • Full GC：使用 serial old 进行 Full GC，单线程，非常耗时。

3.3. 方法区的回收

主要是对常量池的回收和对类的卸载。

类的卸载条件：

  • 该类所有的实例都已经被回收，此时堆中不存在该类的任何实例。
  • 加载该类的 ClassLoader 已经被回收。
  • 该类对应的 Class 对象没有在任何地方被引用，即无法在任何地方通过反射访问该类方法。
